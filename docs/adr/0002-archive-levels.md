# ADR-0002：归档等级 L0 - L2R 划分

- 日期：2026-06-06
- 状态：Accepted
- 影响范围：数据模型、Worker、合规

## 背景

不同来源的媒体内容可获取程度差异极大：

- 公开直链 / 用户上传：能拿到真实原始文件
- 平台流式视频（抖音、B 站、小红书）：浏览器播放时能从网络流抓回完整资源，但不是平台明示的下载入口
- 平台动态渲染内容（图集、登录墙后的帖子）：只能视觉复刻
- 复杂动态页面：可能只能拿到 HTML 快照

如果不区分这些层次：

- 用户分不清"这个文件是不是真原始"
- 后续 AI 处理无法判断保真度
- 合规边界混乱（录屏可能被误标为原始文件）

## 可选方案

### A. 不分层，一个 `archive_level` 字段（low/mid/high）
- 优点：简单
- 缺点：语义模糊，"high" 对一段录屏和一份原始 MP4 含义完全不同

### B. 分两层：metadata-only 与 with-asset
- 优点：清晰简单
- 缺点：把录屏和原始文件等同视为"有资产"，掩盖合规和保真度差异

### C. 分五层 L0 / L1 / L2-O / L2-N / L2-R（选定）
- 优点：原始 / 网络流 / 视觉复刻互斥可识别，合规边界清晰
- 缺点：用户学习曲线略陡（但内部用户只有自己）

## 决策

采用方案 C 五层划分：

```text
L0    URL + 元数据
L1    L0 + HTML/PDF/截图快照
L2-O  L1 + 真实原始文件          asset_source = original_file
L2-N  L1 + 浏览器合法请求的媒体流  asset_source = network_capture
L2-R  L1 + 屏幕录制 / 高保真截图  asset_source = visual_recording / screenshot_*
```

Item 的 `archive_level` 字段记录"本 Item 当前实际达到的最高等级"。同一 Item 多个 Snapshot 可能等级不同，取最高。

## 理由

- **L2-O 必须严格**：只有真实原始文件才配此标签；录屏、抓流都不行。否则后续 AI 训练或导出会被污染数据误导
- **L2-N 独立成档**：网络流抓取在画质和文件大小上远优于录屏，但来源是平台流分发而非用户上传或平台明示下载入口，"既不是原始也不是录屏"必须有自己的位置
- **L2-R 兜底**：当 L2-N 失败（DRM、加密、签名失效）或不适用（图集、直播）时降级。视觉复刻是最稳的兜底路径
- **L0 / L1 全员必备**：即使采集失败，至少保留来源和快照

## 影响

- 数据：`Item.archive_level` 枚举字段；`Asset.asset_source` 枚举字段，二者一致性需校验
- Worker：Pre-flight 决策树明确按 L2-O > L2-N > L2-R 优先级走
- UI：详情页用图标 / 颜色区分不同等级，让用户一眼分辨
- 合规：`asset_source` 枚举中不存在"录屏标记为原始"的可能性

## 风险

| 风险 | 缓解 |
|---|---|
| 用户混淆"L2-N 是不是下载" | UI 文案明确"网络流抓取，非平台官方下载" |
| L2-N 边界被滥用走向破解 | 强制：DRM/加密/签名失效必降级，且禁止逆向（README §12） |
| 多 Snapshot 等级回退（新版本反而更低） | Item.archive_level 取所有 Snapshot 的最大值，不回退 |

## 复核条件

- 出现新类型的归档形态（如平台官方 API 导出） → 考虑加 L2-E (export)
- 合规边界发生变化 → 重写本 ADR
